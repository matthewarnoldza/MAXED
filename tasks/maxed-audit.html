TL;DR

The headline result: live AI generation was completely broken — every request 502'd because the JSON schema used minItems:3, which Anthropic's structured output rejects. The app silently fell back to the on-device coach every time, so you would have shipped with a dead OpenRouter integration. That's fixed and verified against the real API. Alongside it, several core-feature correctness bugs (progressive-overload deltas, a surprise permanent ban, a non-functional REFINE, multi-device data loss) and a batch of UX/visual nits are fixed. The app builds clean and the main flows pass an automated browser run.

Fixed — Critical

criticalLive AI never worked — every generation 502'd
lib/schema.ts · category: bug

OpenRouter/Anthropic rejected the request: minItems values other than 0 or 1 are not supported. The client caught the 502 and used the offline generator, so the failure was invisible — the live coach, your OpenRouter key, everything, was dead.

Fix: removed minItems from the schema; enforce “3–8 exercises” in the system prompt instead. Verified: live API now returns a real, tailored plan that respects banned lifts & overload.

Fixed — High

highProgressive-overload “+” loads were stored as absolute kg
store/useApp.ts · liftsToPlan · bug

A lift meaning “last PB + 5 kg” became a flat 5 kg on accept (parseFloat('+5')===5). The headline coaching feature silently produced near-zero loads.

Fix: deltas now resolve against the athlete’s PB (base + Δ); absolute and BW loads handled separately.
highAccepting an AI plan permanently banned exercises
store/useApp.ts · acceptPlan/inferBans · data-loss

A one-off quick-start like “Legs, no back squats” wrote a durable, cloud-synced banned stance for Back Squat — affecting every future workout & AI request, with no UI cue and no easy undo.

Fix: removed the durable-ban side effect from accept (the request is already honoured for that one plan). Bans are now only set explicitly in the Library.
high“REFINE” didn’t refine — it regenerated from scratch
store/useApp.ts · generate · bug

“Swap lift 2 for a cable move” was treated as a brand-new standalone request; the current plan was never sent. The refine plumbing in openrouter.ts was dead from the UI.

Fix: when a plan is on screen it’s sent as the draft so the model edits that plan. Also added a double-submit guard.
highTwo devices (same user) clobbered each other’s logged workouts
app/api/state/route.ts · data-loss

Each device POSTs the whole JSON doc; an unconditional overwrite meant a stale device could wipe the other’s sessions — the irreplaceable training log.

Fix: server now union-merges sessions by timestamp on every save (a stale write can’t drop the other device’s log); a deliberate Reset sends replace:true to clear authoritatively. Verified via API: stale “B” flush returned B+A; reset clears.
highGenerator compound list was stale vs the renamed library
lib/generator.ts · COMPOUNDS · bug / continuity

The library renamed “Bench Press” → “Barbell Bench Press”, so the compound set no longer matched: bench was treated as an isolation lift (wrong set count, no overload), and a redundant accessory pool could emit names absent from the library (forking PB history).

Fix: aligned the compound names to the library and dropped the redundant pool. Verified: bench detected as compound (4 sets).
highOffline coach misread “legs, no back squats” as a Pull day
lib/generator.ts · resolveFocus · bug

The focus resolver matched “back” (from “back squats”) before checking leg keywords, producing a pull workout including Deadlift.

Fix: legs checked first; arms matched on a word boundary so “warm up” can’t false-match. Verified: → Leg Day, excludes Back Squat.

Fixed — Medium

medAI status lied: “OFFLINE COACH” even when live AI works
api/workout/generate (GET) · Settings · Assistant · ux

Settings & the Assistant header keyed off the in-app key only, so on your Vercel deploy (server env key set) they’d say “offline”/“local” while live AI actually ran.

Fix: added a server-key capability probe; UI shows “LIVE AI” / “COACH AI” when either the server or in-app key is present. Verified.
medRest timer drifted on reload / iOS backgrounding
store/useApp.ts · tickRest · bug

Pure −1/sec decrement; locking the phone (iOS throttles timers) or reloading under-counted by the away-duration.

Fix: wall-clock anchored (restEndsAt); remaining time recomputed from the clock, accurate across background/reload.
medRe-opening the coach showed a stale “PLAN SET ✓” card
components/screens/Assistant.tsx · ux

After accepting/discarding then navigating away, the coach reopened on the old result card, not a fresh prompt — felt broken on second use.

Fix: resets to the prompt view on entry unless a plan is actively pending.

Recommended next (not yet done)

Real, verified findings that are larger features or lower-priority polish — your call on which to take.

high No service worker — installed PWA can’t cold-boot offline
The splash said “OFFLINE READY” and the manifest is standalone, but there’s no service worker, so an installed app with no network won’t load at all (the on-device cache is moot if the shell can’t boot, and fonts come from Google). Do: add next-pwa (or a manual SW) to precache the shell + self-host fonts — or drop the offline implication (I already softened the splash label).
high Exercise-name continuity across AI plans
The live model may return “Bench Press” while your history is under “Barbell Bench Press”, forking the PB/overload bucket. Do: add canonical names to the system prompt and/or a name-normaliser on accept that snaps to the closest library name.
med No way to view / delete / repeat a past session
History is per-exercise only; completed sessions are write-only. A training logger should let you open “what did I do last Tuesday”, delete a mis-logged session, and repeat a workout. Do: add a session log + per-session detail with delete/repeat.
med Silent loading / sync-failure states
loadUser has no spinner and failed cloud writes are swallowed — a user whose saves are 500-ing has zero indication. Do: surface a loading state (keyed on loadingUser) and a sync-failure indicator.
med Offline-created profile can’t sync later (FK)
If /api/auth fails (offline), a local-… profile is created with no users row; later state saves are rejected by the foreign key (silent 500). Do: create the users row on first successful online sync, or block profile creation while offline.
med Other UX gaps
  • Duplicate-name profiles are indistinguishable on Login — add a created-date / last-trained sub-label.
  • First-run shows the canned default plan as “TODAY” with no “starter — personalise it” cue.
  • Bodyweight / 0 kg logging — the logger forces a kg; Pull-up/Plank have no proper mode and a 0 kg set can be logged.
nit Hardening & platform polish
  • No rate-limit on /api/auth and no server-side profile delete (junk rows accrue on a public URL).
  • iOS env(safe-area-inset-*) not applied under viewportFit:cover (notch / home-indicator).
  • Broader keyboard a11y & :focus-visible styling.
  • In-flight guards on logSet/acceptPlan (only generate is guarded).
